# v1.2.5 GraphicalMatrix認証ロックアウト設定化 詳細設計

## 1. 目的

GraphicalMatrixの画像列照合に連続して失敗した場合のロックアウト条件を、
`graphicalmatrix.properties` で変更できるようにする。

現行実装では、以下の値が `GraphicalMatrixVerifyServlet` と
`GraphicalMatrixChangeServlet` に重複してハードコードされている。

```java
private static final int MAX_FAILURES = 5;
private static final long LOCK_MILLIS = 15L * 60L * 1000L;
```

本改修では、この値を共通設定として読み込み、通常ログインと変更画面の
GraphicalMatrix照合へ同じ値を適用する。さらに、失敗を繰り返したアカウントには
通常ロックより長い最大ロックを適用する二段階ロックアウトを追加する。

## 2. 設計方針

- 設定元は
  `/opt/shibboleth-idp/conf/graphicalmatrix/graphicalmatrix.properties`
  に統一する。
- 通常ロックの既定値は、従来どおり5回、900秒とする。
- 10回目以降の失敗には、最大ロックとして2592000秒（30日）を適用する。
- DB保存とLDAP保存で同じ設定と動作を使用する。
- 通常ログインと、sequence・MFA方式変更前の本人確認に同じ設定を適用する。
- 管理APIのBearer token認証失敗制限とは分離する。
- 従来LDAP変更経路のLDAPパスワード認証レート制限とは分離する。
- 設定値が不正な場合は、安全でない値へフォールバックせず設定検査を失敗させる。
- DB schemaおよびLDAP schemaは変更しない。

## 3. 対象範囲

### 3.1 設定を適用する処理

| 処理 | 実装箇所 | 適用 |
| --- | --- | --- |
| SPログイン時のGraphicalMatrix画像列照合 | `GraphicalMatrixVerifyServlet` | 対象 |
| IdP自己管理フローでの現在画像列確認 | `GraphicalMatrixChangeServlet` | 対象 |
| 従来LDAP変更経路での現在画像列確認 | `GraphicalMatrixChangeServlet` | 対象 |
| DB保存時の失敗回数・ロック期限更新 | `GraphicalMatrixRepository` | 対象 |
| LDAP保存時の失敗回数・ロック期限更新 | `GraphicalMatrixLdapEnrollmentStore` | 対象 |

DBとLDAPのrepository層は、現在は呼出元から `maxFailures` と `lockMillis` を
受け取っている。二段階ロックアウトでは、失敗後の `failed_count` に応じて
通常ロックと最大ロックを選択する必要があるため、共通の
`GraphicalMatrixLockoutPolicy` を受け取る構造へ変更する。

### 3.2 対象外

以下の設定・処理は本改修では変更しない。

- LDAPパスワード認証自体の失敗回数と制限時間
- TOTPコード認証の失敗制御
- WebAuthn認証の失敗制御
- 管理APIのBearer token認証失敗制御
- challengeの有効期限
- 送信元IP単位のレート制限
- 段階的遅延、指数バックオフ、CAPTCHA

既存の以下の設定は、それぞれ別の保護対象を持つ。

```properties
# 従来変更画面のLDAPパスワード認証
graphicalmatrix.change.ldapRateLimit.failureLimit = 5
graphicalmatrix.change.ldapRateLimit.windowSeconds = 300
graphicalmatrix.change.ldapRateLimit.lockSeconds = 900

# 管理APIのBearer token認証
graphicalmatrix.api.authFailureLimit = 5
graphicalmatrix.api.authFailureWindowSeconds = 60
graphicalmatrix.api.authFailureLockSeconds = 300
```

## 4. 新規設定

`graphicalmatrix.properties` に以下を追加する。

```properties
# 5回目から通常ロック
graphicalmatrix.lockout.failureLimit = 5
graphicalmatrix.lockout.lockSeconds = 900

# 10回目から最大ロック
graphicalmatrix.lockout.maxLockFailureCount = 10
graphicalmatrix.lockout.maxLockSeconds = 2592000
```

| Property | 型 | 既定値 | 許容範囲 | 説明 |
| --- | --- | --- | --- | --- |
| `graphicalmatrix.lockout.failureLimit` | integer | `5` | `1`から`100` | GraphicalMatrix画像列照合をロックするまでの連続失敗回数 |
| `graphicalmatrix.lockout.lockSeconds` | integer seconds | `900` | `1`から`2592000` | 閾値到達時に設定するロック時間。上限は30日 |
| `graphicalmatrix.lockout.maxLockFailureCount` | integer | `10` | `failureLimit + 1`から`1000` | 最大ロックへ移行する保存済み `failed_count` |
| `graphicalmatrix.lockout.maxLockSeconds` | integer seconds | `2592000` | `lockSeconds`から`2592000` | 最大ロック時間。既定値は30日 |

4項目とも `0` は許可しない。`maxLockSeconds=0` を永久ロックとして扱う仕様も
本設計には含めない。無期限ロックは第三者によるアカウント停止攻撃の影響が
大きく、DB・LDAP・CLI・画面で特別な状態管理が必要になるためである。

通常運用では以下を推奨範囲とする。

- `failureLimit`: `3`から`10`
- `lockSeconds`: `300`から`3600`
- `maxLockFailureCount`: `failureLimit + 1`以上、`100`以下
- `maxLockSeconds`: `lockSeconds`以上、最大2592000秒

推奨範囲外でも許容範囲内なら起動可能とするが、設定検査では警告を出す。

## 5. 設定読込

### 5.1 GraphicalMatrixConfig

`GraphicalMatrixConfig` に以下を追加する。

```java
private static final int DEFAULT_LOCKOUT_FAILURE_LIMIT = 5;
private static final int DEFAULT_LOCKOUT_LOCK_SECONDS = 900;
private static final int DEFAULT_LOCKOUT_MAX_LOCK_FAILURE_COUNT = 10;
private static final int DEFAULT_LOCKOUT_MAX_LOCK_SECONDS = 2592000;

private final int lockoutFailureLimit;
private final int lockoutLockSeconds;
private final int lockoutMaxLockFailureCount;
private final int lockoutMaxLockSeconds;
```

公開getterは以下とする。

```java
public int getLockoutFailureLimit()
public int getLockoutLockSeconds()
public long getLockoutLockMillis()
public int getLockoutMaxLockFailureCount()
public int getLockoutMaxLockSeconds()
public long getLockoutMaxLockMillis()
public GraphicalMatrixLockoutPolicy getLockoutPolicy()
```

秒を返すgetterは設定検査と表示に使用し、millisを返すgetterは
ロック期限計算に使用する。

```java
getLockoutLockMillis()    = lockoutLockSeconds * 1000L
getLockoutMaxLockMillis() = lockoutMaxLockSeconds * 1000L
```

`GraphicalMatrixConfig.load()` で以下を読み込む。

```java
final int lockoutFailureLimit = intProperty(properties,
    "graphicalmatrix.lockout.failureLimit", DEFAULT_LOCKOUT_FAILURE_LIMIT);
final int lockoutLockSeconds = intProperty(properties,
    "graphicalmatrix.lockout.lockSeconds", DEFAULT_LOCKOUT_LOCK_SECONDS);
final int lockoutMaxLockFailureCount = intProperty(properties,
    "graphicalmatrix.lockout.maxLockFailureCount",
    DEFAULT_LOCKOUT_MAX_LOCK_FAILURE_COUNT);
final int lockoutMaxLockSeconds = intProperty(properties,
    "graphicalmatrix.lockout.maxLockSeconds",
    DEFAULT_LOCKOUT_MAX_LOCK_SECONDS);
```

プロパティが存在しない場合と、値が空の場合は既定値を使用する。
数値以外の場合と許容範囲外の場合は `IllegalArgumentException` を発生させる。

### 5.2 GraphicalMatrixLockoutPolicy

DB保存とLDAP保存で同じ判定を保証するため、ロック条件の計算を
`GraphicalMatrixLockoutPolicy` に集約する。

```java
public final class GraphicalMatrixLockoutPolicy implements Serializable {
    public LockDecision afterFailure(int failedCount);
}
```

`LockDecision` は少なくとも以下を保持する。

```text
locked       ロック対象か
lockMillis   適用するロック時間
level        NONE / NORMAL / MAXIMUM
```

判定規則:

```text
failedCount < failureLimit
    -> NONE、ロックしない

failureLimit <= failedCount < maxLockFailureCount
    -> NORMAL、lockSecondsを使用

failedCount >= maxLockFailureCount
    -> MAXIMUM、maxLockSecondsを使用
```

repository層とサーブレット層は独自に条件式を持たず、このpolicyの結果を使用する。

### 5.3 設定スナップショット

GraphicalMatrix challenge開始時に読み込んだ `GraphicalMatrixConfig` は、
現在と同じくHTTPセッションへ保存する。challengeの途中で設定ファイルが
変更されても、そのchallengeは開始時の設定値で完了させる。

新しいchallengeから新設定を適用する。複数IdPノードで値が異なると、
ノードによってロック条件が変わるため、全ノードへ同じ設定を配布する。

## 6. 入力値検査

`GraphicalMatrixConfig.validate()` に以下を追加する。

```text
1 <= graphicalmatrix.lockout.failureLimit <= 100
1 <= graphicalmatrix.lockout.lockSeconds <= 2592000
graphicalmatrix.lockout.failureLimit
    < graphicalmatrix.lockout.maxLockFailureCount <= 1000
graphicalmatrix.lockout.lockSeconds
    <= graphicalmatrix.lockout.maxLockSeconds <= 2592000
```

エラーメッセージは設定名と許容範囲を含める。

```text
GraphicalMatrix lockout failure limit must be between 1 and 100.
GraphicalMatrix lockout seconds must be between 1 and 2592000.
GraphicalMatrix maximum lock failure count must be greater than the normal failure limit and at most 1000.
GraphicalMatrix maximum lock seconds must be between the normal lock seconds and 2592000.
```

極端な値を誤って設定した場合を検出するため、
`GraphicalMatrixConfigCheckTool` は以下の条件で警告する。

- `failureLimit < 3`
- `failureLimit > 10`
- `lockSeconds < 300`
- `lockSeconds > 3600`
- `maxLockFailureCount > 100`

正常時の出力例:

```text
OK: [config] GraphicalMatrix lockout valid: failure_limit=5 lock_seconds=900 max_lock_failure_count=10 max_lock_seconds=2592000
```

警告例:

```text
WARN: [config] GraphicalMatrix lockout failure limit is higher than recommended: failure_limit=20
```

許容範囲外の場合は `CONFIG_CHECK_FAILED` とし、認証開始前の修正を要求する。

## 7. 認証処理の変更

### 7.1 通常ログイン

`GraphicalMatrixVerifyServlet` の `MAX_FAILURES` と `LOCK_MILLIS` を削除する。

repository呼出しには個別の閾値ではなく、設定から生成した共通policyを渡す。

```java
result = repository.verify(
    user,
    selected,
    displayOrder,
    now,
    config.getLockoutPolicy(),
    config.isOrderedSelectionRequired(),
    config.isDuplicateSelectionsAllowed()
);
```

通常ロックに達する前の再試行メッセージは、同じ `failureLimit` を使用する。

```text
remaining = config.getLockoutFailureLimit() - failedCount
```

`failedCount >= failureLimit` では失敗直後にロックされるため、最大ロックまでの
残り回数は認証画面へ表示しない。最大ロックへの移行条件は管理者向け文書と
監査ログで確認できるようにする。

ロック画面へ渡す期限は、サーブレットで `now + lockSeconds` を再計算しない。
DBまたはLDAPへ保存した正確な `locked_until` をrepositoryの結果から取得する。

### 7.2 変更画面

`GraphicalMatrixChangeServlet` の `MAX_FAILURES` と `LOCK_MILLIS` を削除する。

`verifyForSequenceChange()`、残り回数表示、ロック画面の期限に、通常ログインと
同じpolicyを使用する。

これにより、SPログインで4回失敗したユーザーが変更画面でさらに1回失敗した場合も、
同じ保存済み `failed_count` によりロックされる。認証経路ごとの別カウンタにはしない。

### 7.3 repository層

以下のメソッドが受け取る `maxFailures` と `lockMillis` を
`GraphicalMatrixLockoutPolicy` に置き換える。

- `GraphicalMatrixRepository.verify()`
- `GraphicalMatrixRepository.verifyForSequenceChange()`
- `GraphicalMatrixLdapEnrollmentStore.verify()`
- `GraphicalMatrixLdapEnrollmentStore.verifyForSequenceChange()`

失敗回数を1増加させた後に、必ずpolicyへ新しい `failed_count` を渡す。

```java
final int failed = failedCount + 1;
final LockDecision decision = lockoutPolicy.afterFailure(failed);
final long newLockedUntil = decision.isLocked()
    ? now + decision.getLockMillis()
    : 0L;
```

DB repositoryは行ロックを取得したトランザクション内で判定と更新を行う。
LDAP storeは対象entryを読み取った後、同じpolicyの結果で失敗回数属性と
ロック期限属性を更新する。DB用とLDAP用に条件式を複製しない。

### 7.4 照合結果

`GraphicalMatrixVerifyResult` に、ロック時の正確な期限を保持できる
`lockedUntil` を追加する。

```java
public static GraphicalMatrixVerifyResult locked(
    String auditDetail, long lockedUntil)

public long getLockedUntil()
```

既存の `locked(String auditDetail)` は他の認証処理との互換性のため維持し、
`lockedUntil=0` とする。

既にロック中の場合は保存済みの期限を返す。今回の失敗でロックした場合は、
今回保存した `newLockedUntil` を返す。これにより、通常ロックと最大ロックの
どちらでも画面表示と保存値が一致する。

## 8. 失敗回数とロックの状態遷移

### 8.1 二段階ロック

既定値での状態遷移は以下とする。

| 失敗後の `failed_count` | 判定 | `locked_until` |
| --- | --- | --- |
| `1`から`4` | `NONE` | `0` |
| `5`から`9` | `NORMAL` | 失敗時刻から900秒後 |
| `10`以上 | `MAXIMUM` | 失敗時刻から2592000秒後 |

失敗時:

```text
failed_count = failed_count + 1
decision = lockoutPolicy.afterFailure(failed_count)

decision = NONE:
    locked_until = 0

decision = NORMAL:
    locked_until = 現在時刻 + lockSeconds

decision = MAXIMUM:
    locked_until = 現在時刻 + maxLockSeconds
```

成功時:

```text
failed_count = 0
locked_until = 0
```

ロック中:

```text
locked_until > 現在時刻の場合は照合を実行せずLOCKEDを返す
```

DB保存では行ロックを取得して更新する。LDAP保存では対象ユーザーの属性を更新する。
保存先によって設定値や判定条件を変えない。

### 8.2 ロック期限経過後のfailed_count

現行実装では、`locked_until` の期限が経過しただけでは `failed_count` を0へ戻さない。

既定値の場合、5回失敗して通常ロックされた後、期限経過後の最初の認証が
成功すれば0へ戻る。失敗した場合は `failed_count=6` となり、再び15分ロックされる。
同様に失敗を繰り返して10回目に達すると30日ロックされる。

30日ロックの期限経過後も `failed_count` は10以上のままである。このため、
最初の認証に失敗すると、その失敗時刻から再び30日ロックされる。成功した場合は
`failed_count=0`、`locked_until=0` となり、次回から再び4回まで失敗できる。

本改修はハードコード値の設定化を目的とするため、この動作は変更しない。
期限経過時の自動リセットや失敗回数の観測時間窓は、別の認証仕様変更として扱う。

`failed_count` と `locked_until` が0へ戻る主な条件は以下である。

- GraphicalMatrix画像列照合に成功したとき
- sequence変更前の現在画像列確認に成功したとき
- 管理CLIでunlockまたは失敗回数リセットを実行したとき
- RESET、MFA方式変更、再プロビジョニングなど、既存実装が認証状態を初期化するとき

### 8.3 監査ログ

画像列照合失敗時の監査detailには、少なくとも以下を含める。

```text
failed_count=<count>,lock_level=<none|normal|maximum>,lock_seconds=<seconds>,locked_until=<epoch_millis>
```

`lock_level=none` の場合は `lock_seconds=0`、`locked_until=0` とする。
画像列、鍵、キーワードなどの秘密情報は記録しない。最大ロック発生を監視側で
識別できるため、将来の通知機能でも同じ `lock_level=maximum` を使用する。

## 9. 既存環境との互換性

### 9.1 設定ファイル

既存の `graphicalmatrix.properties` に新設定がなくても、既定値として
`failureLimit=5`、`lockSeconds=900`、`maxLockFailureCount=10`、
`maxLockSeconds=2592000` を使用する。

第1回から第9回までの失敗に対する動作はv1.2.4以前と同じである。第10回以降は
従来の15分ロックではなく30日ロックとなるため、完全な動作互換ではない。
この変更はv1.2.5のセキュリティ仕様としてUPGRADE.mdへ明記する。

配布物の `graphicalmatrix.properties.idpnew` には新設定を明示する。
既存設定ファイルはインストーラで上書きせず、管理者が差分を確認して追加する。
4項目は相互依存するため、値を変更するときは一組として確認する。

### 9.2 保存データ

以下は変更しない。

- DBの `failed_count`
- DBの `locked_until`
- LDAPの失敗回数属性
- LDAPのロック期限属性
- unlock、RESET、MFA方式変更時の既存リセット処理

DB migrationおよびLDAP schema更新は不要である。

設定変更時の既存状態は次のように扱う。

- `failureLimit` を下げても、設定変更だけでは即時ロックしない。次回失敗時に新閾値で判定する。
- `failureLimit` を上げても、既存の `failed_count` は維持する。
- `lockSeconds` を変更しても、既に保存済みの `locked_until` は変更しない。
- `maxLockFailureCount` を下げても、設定変更だけではロック期限を延長しない。次回失敗時に新閾値で判定する。
- `maxLockFailureCount` を上げても、既存の `failed_count` は維持する。
- `maxLockSeconds` を変更しても、既に保存済みの `locked_until` は変更しない。
- 新しい通常・最大ロック時間は、設定変更後に発生した新規ロックから適用する。

## 10. セキュリティ上の注意

- `failureLimit` を大きくするとオンライン推測に使える試行回数が増える。
- `failureLimit` を小さくすると、第三者による意図的なアカウントロックが容易になる。
- `lockSeconds` を短くすると総当たり耐性が下がる。
- `lockSeconds` を長くすると正規ユーザーへの影響と管理者の解除作業が増える。
- `maxLockFailureCount` を小さくすると最大ロックへ到達しやすくなり、アカウント停止攻撃の影響が増える。
- `maxLockSeconds` の既定値は30日であるため、最大ロックを検知し、本人確認後に管理CLIで解除できる運用を用意する。
- `maxLockSeconds=0` を永久ロックとしては扱わず、不正な設定として拒否する。
- このロックはユーザー単位であり、送信元IP単位の大量アクセス制御ではない。
- 大量アクセス対策は、IdP前段のリバースプロキシやWAFでも実施する。
- 複数IdPノードでは、設定値と時刻同期を一致させる。

## 12. テスト設計

### 12.1 設定単体テスト

新しい `GraphicalMatrixLockoutConfigTest` を追加し、以下を確認する。

1. 設定なしで `5`、`900`、`10`、`2592000` になる。
2. `failureLimit=3`、`lockSeconds=120`、`maxLockFailureCount=6`、`maxLockSeconds=3600` を読み込める。
3. 空文字は既定値になる。
4. 数値以外を拒否する。
5. `failureLimit=0` と `101` を拒否する。
6. `lockSeconds=0` と `2592001` を拒否する。
7. `maxLockFailureCount <= failureLimit` と `1001` を拒否する。
8. `maxLockSeconds < lockSeconds`、`0`、`2592001` を拒否する。
9. 秒を返す値が正しくミリ秒へ変換される。
10. policyが失敗回数に応じて `NONE`、`NORMAL`、`MAXIMUM` を返す。
11. 管理APIの `authFailureLimit` を設定してもGraphicalMatrixロック値は変わらない。
12. LDAP変更経路の `ldapRateLimit.failureLimit` を設定してもGraphicalMatrixロック値は変わらない。

### 12.2 DB repositoryテスト

テスト用DBで `failureLimit=3`、`lockSeconds=120`、
`maxLockFailureCount=6`、`maxLockSeconds=3600` のpolicyを使用する。

1. 1回目と2回目の失敗は `FAIL` になる。
2. 3回目から5回目の失敗は通常の `LOCKED` になる。
3. `locked_until` が `now + 120000` になる。
4. 6回目以降の失敗は最大の `LOCKED` になる。
5. 最大ロックの `locked_until` が `now + 3600000` になる。
6. 結果オブジェクトの `lockedUntil` がDB保存値と一致する。
7. ロック中は正しい画像列でも `LOCKED` になる。
8. ロック期限経過後に正しい画像列を入力すると成功し、カウンタが0になる。
9. 最大ロック期限経過後に失敗すると、再び最大ロックになる。
10. 通常ログインとsequence変更確認で同じカウンタを使用する。

### 12.3 LDAP保存テスト

LDAP統合試験では、DBと同じ通常・最大閾値、ロック期限、成功時リセットを確認する。
属性更新結果として、失敗回数属性とロック期限属性を読み戻す。
同じ入力列に対するDBとLDAPの `LockDecision` と `lockedUntil` が一致することを確認する。

### 12.4 サーブレットテスト

1. 通常ログインの残り回数表示が設定値から計算される。
2. 変更画面の残り回数表示が設定値から計算される。
3. 通常ロックと最大ロックの両方でロック画面へ遷移する。
4. ロック画面の期限がrepositoryから返された保存値と一致する。
5. challenge開始後に設定ファイルを変更しても、そのchallengeは開始時設定を使用する。
6. 新しいchallengeは変更後の設定を使用する。

### 12.5 監査ログテスト

1. ロック前の失敗で `lock_level=none` が記録される。
2. 通常ロックで `lock_level=normal` と適用秒数が記録される。
3. 最大ロックで `lock_level=maximum` と適用秒数が記録される。
4. `locked_until` がDBまたはLDAPの保存値と一致する。
5. 画像列や暗号鍵などの秘密情報が出力されない。

### 12.6 設定検査・配布物テスト

```bash

  --idp-home /opt/shibboleth-idp \
  --config-only
```

以下を確認する。

- 既定値でOKになる。
- 推奨範囲外、許容範囲内ではWARNになる。
- 許容範囲外では `CONFIG_CHECK_FAILED` になる。

## 13. 運用手順

設定変更例:

```properties
# 5回目から通常ロック
graphicalmatrix.lockout.failureLimit = 5
graphicalmatrix.lockout.lockSeconds = 900

# 10回目から最大ロック
graphicalmatrix.lockout.maxLockFailureCount = 10
graphicalmatrix.lockout.maxLockSeconds = 2592000
```

変更後は設定検査を実行する。

```bash
sudo /opt/shibboleth-idp/bin/graphicalmatrix-plugin-check.sh \
  --idp-home /opt/shibboleth-idp \
  --config-only
```

各リクエストで設定ファイルを読み込む構造だが、進行中challengeとの設定差や
複数ノード間の不一致を避けるため、全ノードへ設定を配布した後に
Jettyをローリング再起動する運用を推奨する。設定ファイルだけの変更では
WAR再構築は不要である。

## 14. 受入条件

- ソースコードからGraphicalMatrix画像列照合用の `MAX_FAILURES=5` と
  `LOCK_MILLIS=15分` が削除されている。
- 新設定未記載時は、5回目から900秒、10回目から2592000秒ロックされる。
- 1回目から4回目はロックされず、5回目から9回目は通常ロックになる。
- 10回目以降は最大ロックとなり、期限経過後の再失敗でも最大ロックになる。
- 正しい画像列で認証すると `failed_count` と `locked_until` が0になる。
- 通常ログインと変更画面が同じ設定値を使用する。
- DB保存とLDAP保存で同じ結果になる。
- ロック画面の期限、監査ログ、DB/LDAP保存値が一致する。
- 管理APIとLDAPパスワード認証のレート制限設定に影響しない。
- 不正値を設定検査で検出できる。
- DB/LDAP schema変更なしでアップグレードできる。
- 単体テスト、repositoryテスト、設定検査、パッケージ検査が成功する。
