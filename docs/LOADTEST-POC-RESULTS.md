# GraphicalMatrix認証負荷試験結果（ローカルPoC）

## 1. 目的と評価範囲

本書は、GraphicalMatrix を含む2FAS-KW認証フローについて、ローカルの検証環境で実施した
同時認証PoCの条件、結果、および再実施手順を記録するものである。

この試験は、IdP、LDAP、PostgreSQL、SimpleSAMLphpテストSP、および負荷生成プログラムを
同一ホストに配置した構成である。そのため、本結果はこの同居構成での総合的な挙動を示すものであり、
IdP単体または本番環境の最大処理能力を保証するものではない。本番相当の性能評価では、
負荷生成ホストをIdPとは分離すること。

## 3. テスト内容

### 3.1 認証フロー

負荷生成プログラムは、画面だけを模擬するのではなく、以下の実際の認証経路を使用する。

```text
負荷生成プログラム
  -> SimpleSAMLphp テストSP
  -> Shibboleth IdP の Password 認証（LDAP）
  -> GraphicalMatrix challenge の表示
  -> 25枚のタイル画像取得と4件の選択送信
  -> GraphicalMatrix照合（PostgreSQL）
  -> SAML Response
  -> SimpleSAMLphp ACS
  -> 認証完了
```

試験用にLDAPユーザー1,000件と、対応するGraphicalMatrix登録情報を事前登録した。
各ユーザーは同一の4件の選択順を持つ。各認証フローでは25枚のタイル画像を取得する。
画像リクエストには`Cache-Control: no-cache`を指定するため、ブラウザキャッシュの効果を
前提にしない測定である。

### 3.2 測定項目

| 指標 | 内容 |
| --- | --- |
| `password_to_matrix` | Password送信からGraphicalMatrix challenge表示までの時間 |
| `tile_images` | 25枚のタイル画像を取得する時間 |
| `verify_to_sp` | 選択送信後、SPへ戻って認証が完了するまでの時間 |
| `total` | 認証開始からSPでの完了までの総時間 |
| throughput | 成功認証数を試験全体の経過時間で割った値 |

## 4. 実施フロー

このPoC用スクリプトは製品配布物には含めず、ローカル検証用として管理する。

### 4.1 事前確認

IdP、SP、LDAP、PostgreSQLが起動していることを確認する。

```bash
sudo systemctl is-active \
  jetty-idp.service httpd.service php-fpm.service dirsrv@2faskwtest.service

sudo /opt/shibboleth-idp/bin/graphicalmatrix-plugin-check.sh \
  --idp-home /opt/shibboleth-idp \
  --config-only
```

SimpleSAMLphpが検証するIdP署名証明書と、IdPが実際に使用する署名証明書は一致していなければ
ならない。証明書を再生成または更新した場合は、SP側metadataも更新してから試験する。

### 4.2 試験ユーザーの登録

最初の1回だけ、LDAPユーザー1,000件とDB上のGraphicalMatrix登録情報を作成する。
既存の利用者データを使わず、専用の接頭辞を付けた検証ユーザーだけを使用する。

### 4.3 負荷試験と監視

別の端末で、試験中のCPU、実行待ち、swap、I/O waitを確認する。

```bash
vmstat 1
```

まず50同時認証で基準値を取り、次に100、200へ段階的に増やす。画像リクエストの並列数を
変える場合は、同じ同時認証数で比較する。


### 4.4 判定の流れ

```text
環境・証明書の確認
  -> 試験ユーザーの登録（初回のみ）
  -> 50同時認証の成功率・遅延確認
  -> 100、200同時認証へ段階的に増加
  -> vmstatとresults.jsonを保存
  -> 失敗率、p95/p99、swap、I/O waitを比較
  -> 負荷生成を別ホストへ分離して再測定
```

認証失敗、swapの発生、I/O waitの継続的な上昇、またはp95/p99の急増が現れた場合は、
同時認証数を増やさずに原因を切り分ける。Jettyのthread poolだけを先に増やすことは避ける。

## 5. 結果（12コア基準試験）

すべての試験で1,000認証中1,000件が成功し、認証失敗は発生しなかった。

| 同時認証数 | 画像並列数 | 成功 / 失敗 | 経過時間 | throughput | total p50 | total p95 | total p99 |
| ---: | ---: | --- | ---: | ---: | ---: | ---: | ---: |
| 50 | 6 | 1,000 / 0 | 22.434 s | 44.575 /s | 996.334 ms | 1,114.360 ms | 1,143.897 ms |
| 100 | 6 | 1,000 / 0 | 22.247 s | 44.950 /s | 1,910.318 ms | 2,199.086 ms | 2,214.977 ms |
| 200 | 6 | 1,000 / 0 | 22.329 s | 44.786 /s | 3,726.138 ms | 4,403.784 ms | 4,479.503 ms |
| 50 | 3 | 1,000 / 0 | 21.992 s | 45.471 /s | 971.822 ms | 1,091.759 ms | 1,109.487 ms |
| 100 | 3 | 1,000 / 0 | 21.953 s | 45.552 /s | 1,880.409 ms | 2,174.875 ms | 2,206.390 ms |

認証処理の内訳として、画像取得が最も大きい割合を占めた。以下は画像並列数6の比較である。

| 同時認証数 | password_to_matrix p95 | tile_images p95 | verify_to_sp p95 |
| ---: | ---: | ---: | ---: |
| 50 | 158.452 ms | 865.074 ms | 75.303 ms |
| 100 | 238.428 ms | 1,766.542 ms | 76.180 ms |
| 200 | 416.725 ms | 3,593.315 ms | 81.171 ms |

200同時認証時の`vmstat`では、CPU idleはおおむね41%から86%で推移し、swap in/outと
I/O waitはいずれも0だった。実行待ちプロセス数は一時的に12 CPUを上回り、context switchは
毎秒およそ9万から12万程度まで増加した。メモリの空き容量は小さくなったが、swapの使用はなかった。

## 6. CPU 4コア制限での追試

CPU利用可能数を4コアに制限し、画像並列数6、試験ユーザー1,000件という条件で追試した。
100同時認証および200同時認証は、それぞれ3回ずつ実施した。ホスト上のコンポーネント配置と
認証フローは、12コア基準試験から変更していない。

| 同時認証数 | 試行回数 | 成功 / 失敗 | throughput 中央値 | throughput 範囲 | total p50 中央値 | total p95 中央値 | total p99 中央値 |
| ---: | ---: | --- | ---: | ---: | ---: | ---: | ---: |
| 100 | 3 | 3,000 / 0 | 49.606 /s | 48.380 - 49.786 /s | 1,356.304 ms | 2,078.339 ms | 2,272.928 ms |
| 200 | 3 | 3,000 / 0 | 56.375 /s | 56.204 - 58.084 /s | 2,500.390 ms | 3,420.234 ms | 3,661.728 ms |

個別試行の結果は以下のとおりである。

| 同時認証数 | 試行 | 経過時間 | throughput | total p50 | total p95 | total p99 |
| ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| 100 | 1 | 20.670 s | 48.380 /s | 1,356.304 ms | 2,062.539 ms | 2,272.928 ms |
| 100 | 2 | 20.159 s | 49.606 /s | 1,353.427 ms | 2,253.935 ms | 2,395.929 ms |
| 100 | 3 | 20.086 s | 49.786 /s | 1,423.811 ms | 2,078.339 ms | 2,246.066 ms |
| 200 | 1 | 17.792 s | 56.204 /s | 2,510.773 ms | 3,365.189 ms | 3,474.807 ms |
| 200 | 2 | 17.216 s | 58.084 /s | 2,466.260 ms | 3,420.234 ms | 3,661.728 ms |
| 200 | 3 | 17.738 s | 56.375 /s | 2,500.390 ms | 3,422.843 ms | 3,897.351 ms |

4コア制限下でも全6,000認証が成功し、100および200同時認証のthroughputはそれぞれ安定した。
一方で、200同時認証では100同時認証に比べてthroughputの中央値が約14%向上するのに対し、
total p95の中央値は約65%増加した。通常利用時の応答時間を重視する場合は100同時認証を
安定ライン、200同時認証は高負荷ラインとして扱う。

50同時認証の4コア測定は単発であり、キャッシュ状態などの影響を除外できないため、
この追試の結論には用いていない。4コアと12コアの性能差を評価する場合は、各同時認証数を
同じ手順で複数回測定し、中央値で比較する必要がある。

## 7. CPU 1コア・4 GiBでの追試

CPU利用可能数を1コア、メモリを4 GiBに制限し、画像並列数6、試験ユーザー1,000件の条件で
追試した。10、25、50、100、200同時認証をそれぞれ3回実施した。認証フローおよび同居する
コンポーネントは、前述の試験から変更していない。

| 同時認証数 | 試行回数 | 成功 / 失敗 | 成功率 | throughput 中央値 | throughput 範囲 | total p95 中央値 | total p99 中央値 |
| ---: | ---: | --- | ---: | ---: | ---: | ---: | ---: |
| 10 | 3 | 3,000 / 0 | 100% | 16.254 /s | 11.177 - 19.468 /s | 543.666 ms | 566.885 ms |
| 25 | 3 | 3,000 / 0 | 100% | 15.678 /s | 13.650 - 15.678 /s | 1,291.029 ms | 1,357.474 ms |
| 50 | 3 | 3,000 / 0 | 100% | 15.964 /s | 15.658 - 18.134 /s | 2,459.805 ms | 2,730.024 ms |
| 100 | 3 | 2,963 / 37 | 98.77% | 13.563 /s | 13.047 - 16.016 /s | 5,695.746 ms | 6,405.433 ms |
| 200 | 3 | 2,160 / 840 | 72.00% | 16.621 /s | 15.842 - 17.552 /s | 9,100.694 ms | 9,707.430 ms |

100および200同時認証の`total`統計値は、成功した認証フローだけを対象とする。失敗した認証の
利用者影響は表の遅延値には含まれないため、成功率と合わせて評価する必要がある。

100同時認証以上で、IdP処理ログに以下のLDAP接続プール待機タイムアウトが記録された。

```text
org.ldaptive.pool.BlockingTimeoutException:
Block time of PT3S exceeded waiting for connection on pool bind-pool with max size of 10
```

これは資格情報の不一致ではなく、IdPのLDAP Password認証用bind poolが最大10接続まで使用中で、
3秒以内に空き接続を取得できなかったことを示す。`IdP password authentication was rejected`は
この例外を受けた試験プログラム側の結果表示である。

1コア構成では、10、25、50同時認証の全9,000認証が成功した。通常利用の応答時間を重視する
場合は25同時認証を推奨ライン、50同時認証は高負荷ラインとする。100同時認証では失敗が発生し、
200同時認証では失敗率が28%に達したため、運用対象外とする。

10同時認証は各フローのp95が約0.5秒で安定する一方、throughputには大きなばらつきがある。
これは負荷生成プログラムも同じ1コア上で動作している影響を受けるためである。IdP単体の性能を
確定するためには、負荷生成ホストを分離した再試験が必要である。

### 7.1 メモリ増量の影響

本追試ではメモリを4 GiBへ増量したが、100同時認証以上で発生した認証失敗の直接原因は、
LDAP bind poolの接続待機タイムアウトである。したがって、メモリを増量してもCPU処理時間や
LDAP bind poolの最大接続数、待機時間は変わらず、この失敗を直接解消する対策にはならない。

メモリ増量が有効になり得るのは、swap in/outの発生、Java GCの長時間停止、OOM、PHP-FPMの
再起動、またはページキャッシュ不足によるI/O待ちが確認された場合である。これらがなければ、
4 GiBから8 GiB以上へ増量しても、throughputの大幅な増加は期待できず、改善しても主にp99の
ばらつき軽減に留まる可能性が高い。

メモリだけの影響を測る場合は、CPUを1コア、画像並列数を6に固定し、メモリ容量だけを変更して
25、50、100同時認証を各3回実施する。試験中は以下を記録する。

```bash
vmstat 1
free -h
```

`vmstat`の`si`と`so`が0で、`wa`も継続的に低ければ、メモリは主な制約ではない。1コア構成で
100同時認証を成立させるには、メモリ増量よりCPU増強、またはLDAP bind poolの接続数・待機時間の
設計見直しを優先する。

## 8. 評価

以下は12コア基準試験に関する評価である。1コア構成の実用域は前節のとおりであり、
12コア基準試験の結果をそのまま適用してはならない。

- 12コアの同居構成では約45成功認証/秒を維持し、1,000件の連続認証で失敗は発生しなかった。
- 同時認証数を50から200へ増やしてもthroughputはほぼ一定である一方、total p95は約1.1秒から
  約4.4秒へ増加した。これは、処理能力の上限付近では待ち行列により応答時間が増えることを示す。
- `verify_to_sp`のp95は約75 msから81 msでほぼ一定だった。DB上のsequence照合と、
  SAML応答をSPへ戻す区間は、この試験における主な待ち時間ではない。
- 25枚のタイル画像取得が認証時間の大半を占める。200同時認証かつ画像並列数6では、理論上
  最大1,200件の画像リクエストが同時進行し得るため、Webコンテナ、HTTP接続、スケジューリングの
  影響を受けやすい。
- 画像並列数を6から3へ下げてもthroughputは大きく変化しなかった。この範囲では、画像並列数だけ、
  またはJettyの`maxThreads`だけが支配的な要因とは判断できない。

このため、現時点ではJettyのthread pool拡大を先行させるのではなく、負荷生成ホストを分離して
IdP側のCPU、メモリ、DB、ネットワークを個別に測定することを優先する。

## 9. 次の試験

本番相当の評価では、次の順に実施する。

1. k6またはJMeterをIdPと別ホストに配置する。
2. SP、IdP、LDAP、PostgreSQLを分離した構成で、同じ認証フローを測定する。
3. 50、100、200同時認証を基準に、成功率、p50、p95、p99、throughputを比較する。
4. IdP、SP、LDAP、PostgreSQLそれぞれのCPU、メモリ、ネットワーク、接続数、GCを記録する。
5. 静的画像をCDNまたは専用Webサーバへ分離する構成も比較し、`tile_images`の待ち時間を評価する。

負荷試験環境を別サーバで構築する一般手順は、[INSTALL_LOADTEST.md](./INSTALL_LOADTEST.md)を参照する。
